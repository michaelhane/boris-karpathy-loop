# Review: v0.2.1 fix-up cluster (reviewer Write tool + 3 deferred nits + version bump)

## Findings

### [CONCERN] Reviewer's Write-tool fix is reasoned, not verified
- **Principle:** 4 (goal-driven execution — verifiable success)
- **Where:** agents/karpathy-reviewer.md:80 (the new prose) and the commit message of 444aaca
- **Why it matters:** The fix is plausible and I literally just exercised the new path successfully while writing this very review, so the Write tool clearly works. But the commit message asserts two specific Windows failure modes (parser-aborted prompt before heredoc executes; 8m45s churn ending in a fallback that points at the previous review file) and the v0.2.1 cluster ships no captured artefact of either failure, no test that re-runs the v0.2 prose against a fixed-size payload to demonstrate the failure, and no smoke-test note. If the real root cause was something else (e.g. an unrelated quoting bug in PowerShell heredoc generation, or a tool-permission issue masquerading as a length issue) the Write tool will appear to fix it because it sidesteps the broken code path entirely — and the broken path will resurface later in any other agent that still uses Bash for multi-KB writes (the karpathy-tutor agent, future agents). Karpathy principle 4 specifically: "Absence of verification is itself a finding."
- **Suggested resolution:** Either (a) capture one reproduction of the v0.2 failure (one Windows session, v0.2 reviewer prose, real review payload, screenshot or log of the parser-abort or the silent fallback) and link it from MEMORY.md so the lesson generalises, or (b) explicitly mark the fix as "reasoned, unverified — trusting that Write replaces the failing path" in the commit message and verification_needed, so the next reviewer doesn't assume it was tested.

### [NIT] Two near-duplicate naming notes that will drift
- **Principle:** 2 (surgical changes — minimum viable diff) and 3 (preserve what works)
- **Where:** commands/setup-graphify.md:25 and commands/diagnose-loop.md:43
- **Why it matters:** The fix correctly addresses the v0.2 nit by adding a "package = graphifyy, binary = graphify" note in each command. But the prose is now duplicated almost verbatim across two files (`> Note on naming: the package on PyPI / installed via uv is **graphifyy** (two y's); the CLI binary on PATH is **graphify** (one y). Don't "fix" one to match the other.`). Two copies of the same sentence reliably drift when one of them gets edited and the other doesn't. The previous reviewer's suggested resolution was "Add one comment near the top of each file" — it accepted duplication as the cheapest fix at the time, and that was the right call for v0.2 (no shared docs surface existed). It is still defensible at v0.2.1, but worth naming as a known maintenance debt so a future "I'll just polish this one phrasing" edit doesn't silently desynchronise.
- **Suggested resolution:** Leave as-is (acceptable duplication for a 2-line snippet) but add a single-line comment in each file pointing at the other, or grep both during any future edit. Not worth introducing an include-mechanism for one sentence.

### [NIT] Version bump not propagated to COMMIT_PLAN.md
- **Principle:** 3 (preserve what works — no silent inconsistency) and 4 (verifiable success)
- **Where:** COMMIT_PLAN.md:271-290 (E5 tag step + Definition-of-done) — referenced from the v0.2.1 chore commit
- **Why it matters:** `plugin.json` is at 0.2.1, but `COMMIT_PLAN.md` still describes the release plan in terms of `v0.2.0` ("E5. Tag v0.2.0", "Definition of done for v0.2.0", "version bumped to 0.2.0"). The chore commit message says "No new commands, no manifest schema changes. Schema validated against B0 gate before commit." — it does not promise to update COMMIT_PLAN, and one could argue COMMIT_PLAN is a historical release-plan document that should not be retroactively rewritten. But the v0.2.1 commit also does not create a v0.2.1 tag (only `v0.2.0` exists per `git tag --list`), so a future maintainer reading "Definition of done … v0.2.0 tag created" alongside `plugin.json: 0.2.1` will reasonably ask: is 0.2.1 considered released? Where is the 0.2.1 tag? Is the Definition-of-Done satisfied? This is hidden ambiguity, not a hard bug — Karpathy principle 1 territory if you read COMMIT_PLAN as canonical, principle 3 territory if you read it as historical. Either interpretation deserves an explicit decision recorded somewhere.
- **Suggested resolution:** Pick one: either (a) amend COMMIT_PLAN to add a Phase F for v0.2.1 with its own DoD and tag step, or (b) add one line to README/CHANGELOG making clear that v0.2.x patches are not separately tagged and 0.2.x is rolled into the v0.2 release line. Either option is fine; the cost is the choice itself, not the document churn.
