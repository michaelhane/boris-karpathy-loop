# Review: v0.3.1 — review-gate Stop nudge (the orphan-review trigger)

## Context
This change set adds a second, opt-in review-gate trigger on top of the v0.3.0 merge/push gate: a `Stop`-hook "review nudge". When a turn ends and HEAD carries committed-but-unreviewed must-review changes (HEAD's merge-base-vs-master diff touches a `must_review` path AND no review's `commit_hash` matches HEAD), it emits a soft, debounced `systemMessage`. It is gated behind a new additive `triggers` block (`merge_push` default true, `stop_nudge` default false), debounced once per HEAD via `.claude/review-gate-state.json`, with a cheap Python-free pre-check in the bash `--stop` launcher branch so the per-turn Stop hook only spawns Python for an opted-in project on a new HEAD. The diff is well-scoped: shared helpers were extracted from `evaluate()` (`_resolve_project_dir`, `_bypassed`) and reused, rather than copy-pasted, and the new path mirrors the existing presence-check logic.

I independently confirmed every load-bearing claim: 21/21 tests green; ruff + ty clean on `hooks/` + `tests/`; `claude plugin validate .` passes; `bash -n` clean on the launcher; the launcher fails open with no `$CLAUDE_PROJECT_DIR` and no config; the Stop hook emits `systemMessage` only even with `mode: block` set (never blocks); a v0.3.0 config with no `triggers` key keeps the merge gate firing AND keeps the nudge silent; and the debounce fires at both the bash pre-check and the Python level.

## Findings

### [CONCERN] `_stop_diff_files` fallback `break` is misplaced — a usable `origin/main` can be shadowed by a stale `origin/master`
- **Principle:** 3 (preserve-what-works correctness) / the user's flagged area (a)+(b)
- **Where:** hooks/review_gate.py:534
- **Why it matters:** In the origin fallback loop the `break` (line 534) sits inside the `if ref-exists` block but OUTSIDE the `if cand and cand != head` block. So the loop stops at the first `origin/<mb>` that merely *exists*, not the first that yields a *usable* base. If `origin/master` exists but its merge-base with HEAD equals HEAD (a stale ref sitting at-or-below HEAD), `base` stays `None`, the loop breaks, and `origin/main` — which would give a real base — is never tried. The function then returns `[]` and the nudge is silently suppressed. I reproduced this: with no local master/main resolvable, `origin/master` at HEAD and `origin/main` behind HEAD, the `--stop` run is silent when it should nudge. Note the asymmetry: the FIRST (local) loop at lines 522-526 has the correct structure (it only breaks on a usable base, so it tries every `master_branches` entry); only this fallback loop regresses that. The first loop is the common path, so impact is limited to the rare case where no local master/main exists AND a stale `origin/master` shadows the real line. Failure direction is fail-SILENT (a missed soft nudge), never a block or a nag — which is why this is a CONCERN, not a BLOCKER. But it is a genuine logic defect, not a judgment call, and the fix is one line (move `break` inside the `if cand and cand != head` block, matching the first loop).
- **Suggested resolution:** Make the fallback loop's `break` conditional on finding a usable base, identical to the first loop — i.e. only break when `cand and cand != head`. Add a regression test for the stale-`origin/master`-shadows-`origin/main` topology so the two loops can't drift again.

### [NIT] Debounce writes state BEFORE the decision — correct for "once per commit", but undocumented as a deliberate trade
- **Principle:** 1 (surface what was glossed over)
- **Where:** hooks/review_gate.py:565-569
- **Why it matters:** `evaluate_stop` records `last_evaluated_head` and writes state immediately after reading HEAD, *before* it knows whether the nudge will fire. This is the right call for the stated contract ("shown once per commit") — it guarantees the nudge can never nag on subsequent turns of the same HEAD. I verified the inverse cost: if the very first Stop on a HEAD happens to be out-of-scope/reviewed/bypassed (silent), the HEAD is still marked evaluated, so a later in-scope condition on that same HEAD won't nudge. For a soft, opt-in reminder that is the safe direction, and in practice HEAD changes on the next commit. The code comment says "evaluate each HEAD at most once" but doesn't name the deliberate write-before-decide choice or its benign false-negative. A one-line comment would stop a future reader from "fixing" it into a nag.
- **Suggested resolution:** Add a half-line to the debounce comment: state is stamped before the decision on purpose, so a HEAD is surfaced at most once regardless of later same-HEAD turns (false-negative chosen over any chance of nagging).

### [NIT] Launcher `grep -q "$head"` is a coarse substring match on the state file
- **Principle:** 2 (the bash pre-check is intentionally coarse) / user area (b)
- **Where:** hooks/review_gate.sh:23 (the `grep -q "$head" "$state"` debounce pre-check)
- **Why it matters:** The pre-check greps the full 40-char HEAD as an unanchored substring of `review-gate-state.json`. Today the state file only ever contains `{"last_evaluated_head": "<sha>"}`, so this is exact-enough and the authoritative decision is still re-made in Python. But it is a superset check: if the state schema ever grows a second SHA-valued field, a stale match could suppress the Python spawn (fail-silent — the safe direction, but a surprise). The same applies to the `stop_nudge` config grep, which matches the literal anywhere in the JSON including inside the `_comment_triggers` string. Both are fine as written; flagging so the coarseness is a known, intentional property, not an accident — the Python side (`state.get("last_evaluated_head") == head`, `_trigger_enabled`) is correctly exact.
- **Suggested resolution:** No change required. If the state schema ever gains another SHA field, anchor the grep (e.g. `"last_evaluated_head"[[:space:]]*:[[:space:]]*"<head>"`) to match the precision the config grep already uses for `stop_nudge`.

### [NIT] Plan says `argparse selects mode`; implementation uses `"--stop" in sys.argv[1:]`
- **Principle:** 1 / 4 (claim-vs-reality)
- **Where:** hooks/review_gate.py:612 vs COMMIT_PLAN.md Phase K4
- **Why it matters:** COMMIT_PLAN K4 states "`argparse` selects mode". The code uses a plain `"--stop" in sys.argv[1:]` membership test — which is simpler, dependency-light, and entirely adequate for one boolean flag, so the code is the better choice. This is harmless plan-vs-code drift, not a defect, but the DoD prose now describes something the code doesn't do. Worth a one-word fix so the plan stays an accurate record.
- **Suggested resolution:** Edit the K4 line to say the flag is read from `sys.argv` (or drop the `argparse` mention). No code change.

## What was done well
- **The soft-never-blocks contract is airtight.** `evaluate_stop` returns `{"decision": None, ...}` and `emit` only writes `hookSpecificOutput` when `decision` is truthy — I confirmed empirically that even `mode: block` yields a bare `systemMessage` with no `permissionDecision`. The user's #1 worry (any path where the nudge blocks) does not exist.
- **Backward-compat is real, not asserted.** `_trigger_enabled` defaults cleanly when `triggers` is absent or non-dict; a verbatim v0.3.0 config (no `triggers` key) keeps the merge gate firing `ask` and keeps the nudge silent. Tested both directions live.
- **Scope discipline.** Shared logic (`_resolve_project_dir`, `_bypassed`) was extracted and reused by both `evaluate` and `evaluate_stop` instead of duplicated; the merge-gate path is byte-for-byte behaviorally unchanged. The diff is the minimum that adds the trigger.
- **Genuine verification, not "looks right".** 6 new tests cover fires / reviewed-silent / out-of-scope / trigger-off / debounce / merge-gate-off, plus a `run_stop` harness that drives the real subprocess + real git. The non-goal of `SennEnd` is justified against the hooks docs in the plan. Fail-open is consistent across every guard, and the per-turn cost is genuinely bounded by the bash pre-check (config grep → one `git rev-parse` → state grep) before any Python spawn — I confirmed the launcher exits 0 without invoking Python when not enabled / same HEAD / no env.
