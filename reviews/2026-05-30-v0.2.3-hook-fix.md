# Review: v0.2.3 — Windows post-commit graphify hook fix

## Context
The graphify `post-commit` hook is unreliable on Windows/Git-Bash. Two confirmed causes: (1) `command -v graphify` returns the extension-less launcher path while the real launcher is `graphify.exe` (a PE binary); the hook's interpreter detection only special-cases `*.exe`, so the extension-less path falls through to `head -1` on the binary, producing bash's `ignored null byte in input`; (2) the fallback drives a system `python`/`python3` that cannot import the uv-isolated graphify, so the detached rebuild silently no-ops. The fix is a new re-runnable patcher (`scripts/fix_graphify_hook.py`) that injects a short-circuit after the `GRAPHIFY_BIN=$(command -v graphify …)` anchor: on Windows it resolves the sibling `.exe` and drives `graphify update .` directly (AST-only), bypassing both brittle steps. No-op on macOS/Linux. Plus a version bump (0.2.2→0.2.3), a COMMIT_PLAN Phase H close-out, and one README table row.

I independently confirmed the root cause on this machine: `command -v graphify` → `/c/Users/micha/.local/bin/graphify` (extension-less), with `/c/Users/micha/.local/bin/graphify.exe` present as a sibling; and `~/.cache/graphify-rebuild.log` does **not** exist (the background rebuild has never run here). I also verified `ruff check`, `ruff format --check`, and `ty` all pass, and that `graphify update <path>` and its `--force` flag are real CLI surface.

## Findings

### [CONCERN] Acceptance test is unrun — the exact "dispositioned-away verification" that caused the v0.2.1→v0.2.2 bug
- **Principle:** 4
- **Where:** COMMIT_PLAN.md:512 (the unchecked `- [ ] Acceptance: …` box) and the live hook end-to-end
- **Why it matters:** Every link in the chain is verified *except the one that matters*. I confirmed statically that the load-bearing branch fires on this machine (the `?*` + `[ -e "${GRAPHIFY_BIN}.exe" ]` path, since `command -v` returns the extension-less path here — note this means the `*.exe` branch is **not** the one exercised on the primary target machine; the sibling-resolution branch is). But "the branch is reached" is not "the rebuild ran and the stamp advanced." The prior failure mode (`exit 0` no-op) *also* printed a launch line and *also* appeared to succeed; the only ground truth was that the log file never existed. The DoD itself flags this box open, and `reviews/_index.md` records that the last time a verification was deferred on this exact hook, it concealed a real bug. The fix is plausibly correct — this is not a "the code looks wrong" concern — it is that the success criterion is prose + static reasoning, not a fired hook. Do not check the box until: a real commit fires the hook, no null-byte warning appears, the log is created AND shows a completion line with a node count, and `GRAPH_REPORT.md`'s stamp moves to the new HEAD with zero manual steps.
- **Suggested resolution:** Make the v0.2.3 release commit, then capture the hook stderr, `cat ~/.cache/graphify-rebuild.log`, and re-grep "Built from commit" against the new HEAD; paste the evidence into the DoD before tagging. If the detached rebuild is racy (graph not yet written when you check), tail the log until it reports done rather than sampling the stamp immediately.

### [NIT] Dropping the `GRAPHIFY_FORCE` / `--force` path on Windows is a behavior change, but a defensible one — note it explicitly
- **Principle:** 3
- **Where:** .git/hooks/post-commit:35-42 (injected block) vs. :95-96 (original `_rebuild_code(force=_force)`)
- **Why it matters:** The original inline rebuild honored `GRAPHIFY_FORCE=1` → `_rebuild_code(force=True)` (overwrite the graph even if the rebuild has fewer nodes). The injected Windows path calls `graphify update .` with no `--force`, so on Windows that escape hatch is silently gone. I checked: `GRAPHIFY_FORCE` is referenced **nowhere** in this repo or its docs — it is an undocumented affordance baked into the graphify-generated hook, not part of this plugin's contract — and `graphify update` exposes `--force` as a documented flag, so the capability is trivially recoverable. That is why this is a nit, not a concern. But it is a real, unannounced behavior delta on the platform the fix targets, and Principle 3 says surface it rather than let it ride silently. The COMMIT_PLAN H1 narrative explains *why* the CLI is driven but does not mention that `--force` was dropped.
- **Suggested resolution:** One line in the patch-block comment (and/or COMMIT_PLAN H1): "drops the undocumented GRAPHIFY_FORCE path on Windows; recover with `--force` if a shrinking rebuild ever needs to overwrite." Optionally honor it: `[ "${GRAPHIFY_FORCE:-}" ] && set -- --force` style. Not required for merge.

### [NIT] `graphify update .` correctness depends on the implicit post-commit CWD — make the assumption legible
- **Principle:** 1
- **Where:** .git/hooks/post-commit:39 (`nohup "$_BKL_GRAPHIFY_EXE" update . …`)
- **Why it matters:** `update .` resolves the target tree from the process CWD. Git runs `post-commit` with CWD = the top of the working tree (I confirmed the worktree root is the repo root), so `.` is correct in the normal case. The original hook made the *same* CWD assumption (it ran the rebuild from CWD too), so this is not a regression — but the choice of `.` over, say, deriving the toplevel via `git rev-parse --show-toplevel` is an unstated decision. An exotic git wrapper or a hook invoked with a different CWD would point `update` at the wrong tree, and because the rebuild is detached the failure would again be silent.
- **Suggested resolution:** Either leave a one-line comment that `.` relies on git's post-commit CWD = worktree root, or harden to `update "$(git rev-parse --show-toplevel)"`. Low priority; flagging the assumption per Principle 1, not demanding the change.

### [NIT] Idempotency marker is robust, but worth a one-line guard-comment about the install→re-patch cycle
- **Principle:** 4
- **Where:** scripts/fix_graphify_hook.py:104 (`if FIX_MARKER in text:`) and the `>>> … <<<` markers in `PATCH_BLOCK`
- **Why it matters:** I traced the idempotency and it is sound: `FIX_MARKER` is checked before anything else and is present in both fence comments, so a second run on a patched hook returns `already` and writes nothing. Double-insert is impossible because the `GRAPHIFY_BIN=$(command -v graphify …)` anchor occurs exactly once in the graphify hook (verified against the live file). The atomic write (`mkstemp` in the same dir → `os.chmod` → `os.replace`, temp unlinked on any exception) means a failed/partial write cannot leave a corrupt hook — good, and `os.replace` is the right call over `os.rename` for the Windows overwrite case. The one non-obvious behavior is the intended `graphify hook install` (strips marker, restores anchor) → re-run patcher (re-inserts once) cycle; it works, but it is the kind of operational contract that rots without a note.
- **Suggested resolution:** None required. Optionally a single comment near `FIX_MARKER` documenting the reinstall→re-run loop so a future reader does not assume the marker alone is the whole idempotency story.

## What was done well
- **Root cause was actually nailed, not guessed.** The COMMIT_PLAN openly records that the brief's original "CRLF / `tr -d '\0'`" hypothesis was *wrong* and replaces it with the empirically reproduced cause — and went further to find the second, larger failure mode (system-python cannot import uv-isolated graphify; rebuild log never existed). That is Principle 1 done right: the prior assumption was surfaced and corrected rather than quietly worked around. I reproduced both signals independently.
- **Shell-safety of the injected block is genuinely careful.** The `case` subject is quoted (no path re-globbing), `${GRAPHIFY_BIN}.exe` and `"$_BKL_GRAPHIFY_EXE"` are quoted, the `update .` args are fixed (no interpolation into an eval'd string), and redirects are quoted. There is no injection vector and space-bearing paths are safe. The block leaves the original detection fully intact below the fence, so macOS/Linux is a true no-op.
- **Fail-loud distinctions are precise.** `not-graphify` (missing `graphify-hook-start`) vs. `no-anchor` (shape drifted) are separated with actionable messages, and both exit non-zero rather than writing anything — exactly the right posture for a tool that mutates an un-versioned file.
- **Surgical and honest scope.** New file + one-line version bump + one README row + plan close-out. No drive-by refactors, no abstraction for a single caller. `--dry-run` and the argparse shape mirror the existing `scripts/*.py`. Lint/format/type all pass (verified).
- **The fix correctly targets the branch that fires on the real machine** (sibling-`.exe` resolution), not just the tidy `*.exe` case — which is the difference between a fix that works and one that only looks right in the comment.
