# Review: v0.3.0 review-gate hook (merge/push-to-master review floor)

## Context
First hook infrastructure for the boris-karpathy-loop plugin: a `PreToolUse`/`Bash` hook that fires when a `git merge`/`git push` is about to land a project-declared *must-review* path on a master branch without a fresh review in `reviews/` (matched by front-matter `commit_hash`). Opt-in per project via `.claude/review-gate.json`; plugin default OFF; warn/ask/block modes; `REVIEW_GATE_BYPASS=1` sanctioned + logged escape; fail-open on every error. Judged against the stated design intent (gate the convergence point, presence-check not quality-check, safe-by-default fail-open), not against an ideal gate. The tracked diff (manifest bump + `hooks` key, COMMIT_PLAN Phase J, README section) is purely additive — no deletions, nothing destroyed (Principle 3 clean).

I verified the suite (11/11 green), `ruff check`/`ruff format --check`/`ty` (all clean — DoD claims honest), `claude plugin validate .` (passes), launcher fail-open on empty stdin (exit 0), and launcher end-to-end with a Windows-style `cwd` (fires `ask` + writes the log line — the DoD's "launcher e2e fired ask + logged" claim holds). The design's hardest call — three-dot `HEAD...ref` for the merge range — is **correct**: I confirmed it isolates exactly what the feature branch introduces even after master diverges, where two-dot would wrongly pull in master-only files. That is the right choice for the parallel-session collision scenario this gate targets.

## Findings

### [CONCERN] `git merge -m/-X/-s <value> feature` is not detected — the value is consumed as the ref
- **Principle:** 1 (glossed-over edge case) / 4 (no test exercises it)
- **Where:** hooks/review_gate.py:117-132 (`positionals`), consumed at hooks/review_gate.py:181-183 (`_detect_merge` takes `refs[0]`)
- **Why it matters:** `positionals()` only knows three arg-eating flags (`-o`, `--repo`, `--push-option`). For `merge`, the common option `-m <msg>` (and `-X <opt>`, `-s <strategy>`) eats its following token, but the gate doesn't know that, so the *value* lands as the first positional. I confirmed: `git merge -m wip feature` → `tip_ref="wip"`. Downstream `git rev-parse --verify wip` fails → `tip_sha is None` → fail-open **silent** (review_gate.py:397-399). `git merge -m "<message>" feature` is the standard explicit-merge form; under it the gate silently does nothing — and a deliberate `-m` merge to master is exactly the kind of intentional convergence the gate exists to catch. (Worse sub-case: if the `-m`/`-X`/`-s` value happens to name a real ref/branch, the diff range is computed against the wrong ref rather than failing open.) Failure direction is under-enforcement, not a false block, so not a blocker — but it is a real, common hole, not a theoretical one. Note `--no-ff`, `--no-edit`, `--strategy-option=theirs` (=-form) all parse correctly; only the space-separated arg-eating flags break.
- **Suggested resolution:** Teach `positionals()` (or a merge-specific arg parse) the arg-eating merge flags (`-m`, `-X`, `-s`, and their long forms `--message`, `--strategy`, `--strategy-option`, `--gpg-sign`/`-S` optional-arg). Add a test for `git merge -m "msg" feature`.

### [CONCERN] Push diff-range hardcodes `origin/` regardless of the actual remote
- **Principle:** 1 (undisclosed assumption baked into a literal)
- **Where:** hooks/review_gate.py:223-225 (`diff_range = f"origin/{dst_branch}..{tip_ref}"`, `needs_ref = f"origin/{dst_branch}"`); also the `{"origin"}` literal at review_gate.py:204
- **Why it matters:** The push path computes the landing set against `origin/<branch>` even when the user pushes to a different remote (`upstream`, `gh`, a fork). I confirmed the false negative: with `origin/master` up to date but pushing new in-scope work to `upstream` (whose tracking ref is stale), `git diff origin/master..HEAD` is **empty** → gate stays silent, while the correct `upstream/master..HEAD` shows the in-scope file. So `git push upstream master` of money-critical code lands with the gate asleep. If `origin` doesn't exist at all, the `needs_ref` verify fails and it fail-opens (acceptable) — the trap is specifically a repo that has both `origin` and another remote at different states. Single-remote `origin` projects (the common case, and the example config's implied setup) are unaffected, which is why this is a CONCERN not a blocker. The README's "fails open on any undeterminable diff" wording does not disclose that the *determinable-but-wrong-remote* diff is silently mis-scoped.
- **Suggested resolution:** Derive the remote from the parsed push args (first positional that isn't a refspec) and build the range/`needs_ref` from `<remote>/<branch>`; fall back to `origin` only when no remote is named. At minimum, document the origin-only assumption in the README "What it does not do" list and the example-config comment. The `{"origin"}` hardcode at line 204 should fold into the same remote-name resolution.

### [NIT] `path_matches` uses `lstrip("./")` — character stripping, not prefix stripping
- **Principle:** 1 (subtle normalization that can mis-scope)
- **Where:** hooks/review_gate.py:234-235
- **Why it matters:** `str.lstrip("./")` removes *all* leading `.` and `/` characters, not a literal `./` prefix. I confirmed `.env` → `env`, `.claude/secrets.py` → `claude/secrets.py`, `...weird.py` → `weird.py`. Because both the file and the pattern are normalized the same way, a `must_review` pattern written without the leading dot (e.g. `env`) would match a real `.env`, and dotfile patterns get their leading dot silently dropped. In practice `must_review` is project-authored and the shipped example uses ordinary `src/`/`scripts/` paths, so the blast radius is small — but for a project that scopes a dotfile path (`.github/workflows/**`, `.claude/**`) this is an incorrect match surface that could fire (or, in `block` mode, deny) on an unintended dotfile. This is the only path I found that could *over*-fire.
- **Suggested resolution:** Strip a literal prefix instead: `f[2:] if f.startswith("./") else f`. Add a dotfile case to the matcher tests.

### [NIT] Tests prove only happy paths — no false-negative parsing coverage
- **Principle:** 4 (verification doesn't exercise the failure surfaces)
- **Where:** tests/test_review_gate.py (11 cases)
- **Why it matters:** The harness is genuinely good — real temp repos, a real bare origin for the push case, drives the gate as a subprocess, asserts the log line. But every parsing case is the canonical form (`git merge feature`, `git push origin master`). None of the three holes above (`merge -m`, non-origin push, dotfile scope) has a test, so the suite's green status reads as broader assurance than it provides. Principle 4 is "tests that exercise the actual change," and the change is fundamentally best-effort command parsing — the parsing variants are the part most worth pinning.
- **Suggested resolution:** Add cases for `merge -m "msg" feature`, `push upstream master` (second remote at a different state), a dotfile `must_review` entry, and a `merge --no-ff feature` positive (cheap, locks in the currently-correct flag handling).

## What was done well
- **Three-dot merge range is the correct, non-obvious call** and is commented with the reason (review_gate.py:185-189). I verified it holds under master divergence — this is the crux of the gate and it's right.
- **Fail-open is genuinely pervasive and layered:** every `git()` returns None on failure, every guard in `evaluate()` returns None, `log_event`/`main` swallow exceptions, and there's a top-level `try/except: pass` backstop with `sys.exit(0)` (review_gate.py:467-471). I could not find a path where an internal error blocks the user.
- **The launcher's no-Python → exit 0 and the relative-script-name CWD trick are sound**, and I confirmed the end-to-end fire+log works with the Windows-style `cwd` Claude Code actually passes. The `||`-can't-retry rationale in the header comment is correct and non-obvious.
- **`ask` carries no `permissionDecision: deny`** so it never silently hard-blocks; bypass is honored in *any* mode and always logged before returning; `warn` emits a `systemMessage` only. The mode→decision mapping matches the stated "warning-first, logged bypass" contract exactly, and the tests assert the log line and `bypassed` flag.
- **`commit_hash` extraction is tolerant in the right ways:** it pulls the SHA out of messy values like `4180cb8 (uncommitted, base HEAD)` and `uncommitted, base HEAD f3fd436` (I tested both — extracts the real base), the prefix match is bidirectional (7-char review vs 40-char tip), and a 6-char-shared-but-different SHA correctly does *not* match. The presence-check is honestly a presence-check, not a smuggled quality gate.
- **DoD claims verify:** ruff/ty clean, `claude plugin validate` passes, 11 tests green, tracked diff is additive only. No claim-without-verification in the parts I could check.

## Resolution (2026-06-01)

Status flipped `open` → `resolved`, dispositioned in the same session as the review (layer-1 same-commit discipline). All four findings fixed in the working diff; the three `verification_needed` items are answered below and locked with regression tests. Every finding was a *false-negative* (under-enforcement, fail-open) — none could falsely block — and the reviewer reproduced each in a real repo before reporting.

- **CONCERN (merge `-m`/`-X`/`-s <val>` missed):** ✅ **fixed.** `_detect_merge` no longer takes `refs[0]` blindly — it picks the last positional that `git rev-parse --verify <c>^{commit}` resolves to, so a `-m`/`-s`/`-X` value (which does not resolve) is skipped and the real ref wins. Robust against *any* value-flag, not an enumerated list. Locked by `test_merge_with_message_flag_fires` (`git merge -m "ship the ledger" feature/ledger` → fires, `tip_ref == feature/ledger`).
- **CONCERN (push diff-range hardcoded `origin/`):** ✅ **fixed.** `_detect_push` derives the remote from the push args (first positional, per `git push <remote> <refspec>` grammar), falling back to the current branch's `@{upstream}` remote, then `origin`; `diff_range`/`needs_ref` build from `<remote>/<branch>`. The incorrect lone-positional-is-refspec branch and the stray `{"origin"}` literal were removed. Locked by `test_push_to_non_origin_remote_fires` (push to `upstream`, no `origin` present → fires).
- **NIT (`lstrip("./")` mangles dotfiles):** ✅ **fixed.** Replaced with `_strip_dot_slash` (strips a literal leading `./` only); `.env` no longer becomes `env`. Locked by `test_dotfile_not_overmatched` (`must_review: ["env"]` + landing `.env` → silent).
- **NIT (happy-path-only tests):** ✅ **fixed.** Added the three regressions above plus `test_messy_commit_hash_in_review_satisfies` (presence-check tolerates `commit_hash: <sha> (uncommitted, base HEAD)`). Suite is now **15 green**; ruff + ty still clean.

The three-dot merge range and the pervasive fail-open (the parts praised) were left untouched.
