# Review: v0.3.2 — two silent-failure shapes added to reviewer Principle 1

## Context
A patch release (0.3.1 → 0.3.2) of the boris-karpathy-loop plugin. The substantive change is a two-bullet addition to the reviewer's own Principle 1 ("Don't assume — surface what was glossed over") in `agents/karpathy-reviewer.md`, capturing two real silent-failure shapes observed in chief-of-staff: (1) error fallbacks that return a valid-looking empty (`[]`/`{}`/`0`) indistinguishable from a legitimately-empty result, poisoning a downstream dedup/diff/aggregation; (2) recognisers that reroute unrecognised input into a default bucket instead of visibly rejecting it (recognition and validation collapsed into one strict gate). The only other change is the manifest version bump. This is a prompt/doc change — no executable code, no runtime error paths, no tests to exercise.

## Findings

### [NIT] The two new bullets break the terse single-clause style of every other "Look for" item
- **Principle:** 2 (minimum viable diff — applied to prose: the addition is larger than the surrounding form invites)
- **Where:** agents/karpathy-reviewer.md:43-44
- **Why it matters:** Every other bullet in all four "Look for" lists is a 5–11 word fragment ("Error paths picked without rationale", "Defaults that change observable behavior without comment"). The two new bullets are ~45 and ~40 words each, each carrying a parenthetical inline definition plus (bullet 1) an em-dash worked example. A reviewer scanning the checklist now hits two paragraph-length entries among one-liners. The *content* is correct and worth keeping; the issue is purely that the compression is inconsistent with the established list, which slightly weakens the at-a-glance scannability the list format exists for. This is the reviewer prompt instructing itself to flag "1000-line solutions to 100-line problems" — mild irony worth noting, not blocking.
- **Suggested resolution:** Optional. If tightened, the load-bearing, greppable hooks to preserve are: `[]`/`{}`/`0`, "dedup pool / diff / aggregation", "default bucket", and the "recognition vs validation" distinction. The worked example could move to a sub-line or be cut without losing the trigger. Leaving as-is is defensible — the examples are what make these actionable rather than abstract.

### [NIT] Manifest version bumped to 0.3.2 but no doc surfaces what 0.3.2 *is*
- **Principle:** 1 (a version-identity change made with no surfaced description) / 4 (no recorded verification of the manifest bump)
- **Where:** .claude-plugin/plugin.json:3 (vs CLAUDE.md:11 "review-gate hook (v0.3.1)"; COMMIT_PLAN.md last DoD = v0.3.1 at lines 801-813)
- **Why it matters:** The version now reads 0.3.2, but COMMIT_PLAN.md (which has explicit Phase J/v0.3.0 + Phase K/v0.3.1 DoD blocks) has no Phase L / v0.3.2 entry, and CLAUDE.md still titles the feature "(v0.3.1)". A future reader diffing manifest history sees a 0.3.2 with no changelog/DoD line explaining it. This is low severity precisely because: (a) README has no per-version changelog section to fall through (only a forward roadmap line at :223), so no established convention was violated; and (b) `claude plugin validate .` passes (verified in this review). It is a traceability gap, not a correctness bug.
- **Suggested resolution:** Either add a one-line v0.3.2 note to COMMIT_PLAN.md (and refresh the CLAUDE.md heading) describing "P1 gains two silent-failure-shape bullets", or consciously decide that prompt-content patches don't warrant a COMMIT_PLAN phase and note that policy once. Surface the choice rather than leaving the version dangling.

## What was done well
- **Principle placement is correct — the central question of the brief.** P1's own header explicitly scopes "error behavior" ("Did it make decisions about edge cases, **error behavior**, naming, or scope without surfacing them?"), and the existing bullets already cover "Error paths picked without rationale" and "Defaults that change observable behavior without comment." A silent fallback-to-`[]` is exactly an unsurfaced error-path + behavior-changing default. P3, by contrast, is about *destroying existing working code* during a change (removed comments, deleted tests, simplified branches, dropped complexity) — it frames things the agent *removes*, whereas these two bullets describe code the agent *writes*. The bullets do not belong under P3. Placement under P1 is right.
- **Genuinely surgical.** Exactly two files, +2/-0 prose lines plus a one-character version bump. No collateral edits, no reflow of adjacent bullets, no scope creep. Nothing was deleted, so Principle 3 has no exposure at all.
- **Greppable and concrete.** Both bullets name matchable artifacts (`[]`/`{}`/`0`, "dedup pool", "default bucket") and encode a reusable conceptual test (the recognition-vs-validation split), so the reviewer can actually pattern-match against real diffs rather than being handed an abstraction.
- **Faithful to source incidents.** The two shapes match the chief-of-staff lessons in MEMORY ("Empty-on-error = silent failure" and the merged-branch/recognition class) without overgeneralising into speculative cases.
- **Patch-level (0.3.2) is the right SemVer call** for additive prompt guidance that changes no interface, hook, or command behavior.

## Resolution (2026-06-13) — both nits addressed in-session (same-commit close)

- **NIT 1 (length drift): fixed.** The two bullets were tightened from ~40–45 words to ~22–28, keeping every load-bearing greppable hook (`[]`/`{}`/`0`, dedup/diff/aggregation, "default bucket", recognition-vs-validation). They remain the two richest bullets in the list — a deliberate, defensible choice for the two least-obvious patterns — but the paragraph-vs-one-liner clash is gone.
- **NIT 2 (untraceable bump): fixed.** COMMIT_PLAN gains a **Phase L — v0.3.2** entry (what/why-P1/source/patch-rationale/DoD). The CLAUDE.md `## review-gate hook (v0.3.1)` heading is **intentionally** left at v0.3.1: it names the review-gate *hook* mechanism, which this release does not touch — bumping it would falsely imply the hook changed. Surfaced as a conscious choice per the nit's own suggestion.
- Both fixes ship in the same commit as this review file → no status-drift window (lifecycle Layer 1).
