# Review: v0.2.4 — reconcile stale-open reviews + close the status-drift loop

## Context
v0.2.4 is a self-review-driven bookkeeping release. Running Phase D4 (dogfood metrics, the new `reviews/_metrics.md`) surfaced a 5th signal — **status-lifecycle drift**: two review files sat `status: open` while their findings were already fixed in code (a *false-open*, mirror of the v0.2.1→v0.2.2 false-close). The diff (a) flips `v0.2-setup-and-diagnose` open→resolved and `v0.2.2-uv-tool-install-fix` open→closed, each with an attributed Resolution footer and a synced `_index.md` line; (b) makes one real code change — caveats the canned `/logout` in the `diagnose-loop.md` Example so it stops contradicting the rule at :39; (c) hardens the loop against recurrence — `loop-bootstrap.md` now treats per-review front-matter `status:` as source of truth and adds a stale-open drift check, and `review-review.md` gains a "Closing the loop" section documenting a three-layer lifecycle; (d) bumps the manifest to 0.2.4 and adds COMMIT_PLAN Phase I. I independently verified every commit attribution against `git show`, `git log -S`, and the current command files rather than trusting the footers.

## Findings

### [CONCERN] N4 is attributed to the wrong commit (false attribution inside a footer whose entire job is accurate attribution)
- **Principle:** 1 (don't assert what wasn't verified) — and the precise failure mode v0.2.4 exists to prevent
- **Where:** reviews/2026-05-07-v0.2-setup-and-diagnose.md:100
- **Why it matters:** The footer reads `N4 (README v0.2 roadmap stale): ✅ fe5ab5c — roadmap updated`. That is factually wrong. `fe5ab5c` never touched README — `git show fe5ab5c --stat -- README.md` is empty, and the commit body lists only the three command-file nits (claude-plugin-list row, naming note, --reinstall→upgrade). The README roadmap was fixed in **`130b720`**, whose own commit message says "Update README.md v0.2 roadmap to reflect what actually shipped" and whose `--stat` shows `README.md | 6 ++++--`. The fix is real and shipped — so this is not a false-close of the finding — but the *commit citation* is incorrect. The whole thesis of v0.2.4 (and the footer's "Lesson") is that attribution must be trustworthy so the index stops lying. A footer that misattributes 1 of 8 findings while preaching attribution discipline undercuts its own credibility, and is exactly the "looks-right, not verified" pattern the loop is supposed to catch. Six of eight citations are exact (C1/C2/C4 → 130b720 verified in the diff; N1/N2 → fe5ab5c verified; N3's chained fe5ab5c→97b6c9a is accurate and honest); this one is not.
- **Suggested resolution:** Change the N4 citation from `fe5ab5c` to `130b720`. (N1/N2 stay `fe5ab5c`; the "nits in fe5ab5c" framing in the footer's opening paragraph then also needs softening, since N4 — a nit — landed in 130b720, not fe5ab5c.)

### [CONCERN] /review-review still enumerates from _index.md, contradicting the source-of-truth doctrine this same change introduces
- **Principle:** 1 (surface the glossed-over decision) / 3 (the change exposed a latent inconsistency it left half-resolved)
- **Where:** commands/review-review.md:5 and :9 vs. :38 (new "Closing the loop" §) and commands/loop-bootstrap.md:9
- **Why it matters:** v0.2.4's central doctrine, stated twice in this diff — loop-bootstrap:9 ("read each review file's front-matter `status:` — this is the source of truth, not `_index.md`'s prose") and review-review:38 ("treats per-review front-matter `status:` as the source of truth") — is that `_index.md` is *derived* and can lie. Yet `/review-review`'s own operative steps were left untouched and still drive off `_index.md`: line 5 "Walk through `reviews/_index.md`, identify findings still marked `status: open`" and step 1 (line 9) "Read `reviews/_index.md` to enumerate review files... For each review file with `status: open`". The prompt states steps 3-5 are "already correct and unchanged" — agreed, the write-back is fine. The defect is upstream: step 1 *finds* the reviews to re-evaluate by reading the very file the new doctrine says is untrustworthy. Concretely, the false-open just fixed was invisible in `_index.md` prose (loop-bootstrap reported "0 open findings" off `_index.md` while front-matter said `open`). If that recurs — `_index` says resolved, front-matter says open — `/review-review` step 1 would skip the file and never reach the now-correct steps 3-5. The catch-up net has a hole on the exact axis v0.2.4 is closing. This isn't pre-existing architecture I'm reaching for: the change *introduced* the front-matter-is-truth rule and documented it inside review-review.md itself, so it created the contradiction.
- **Suggested resolution:** Bring step 1/the intro in line with the doctrine — enumerate review files by globbing `reviews/*.md` and reading each one's front-matter `status:`, using `_index.md` only as a derived convenience to regenerate (step 5), not as the enumeration source. A direction, not a fix.

### [NIT] _metrics.md preserves a superseded "looks unaddressed" hypothesis as if it were a live claim
- **Principle:** 1 (don't leave a glossed-over wrong statement standing)
- **Where:** reviews/_metrics.md:74-77 (the "Decision (D4)" block)
- **Why it matters:** The Decision section says triage should determine "which remain genuinely open — auth-conflict, `GEMINI_API_KEY` validation, chunk-failure detection look unaddressed." That is the *pre-investigation* guess, and it is the opposite of what the investigation found and the footer asserts: all three (C4 auth, C2 GEMINI, C3 chunk) were in fact fixed in `130b720`. The bottom-of-file "Update (2026-05-30)" (lines 88-98) does correct it ("all findings attributed to 130b720/fe5ab5c"), so the file is honest end-to-end for a reader who finishes it — same provenance-preservation pattern the project uses elsewhere. But a reader who stops at the Decision block (the natural action-item section) carries away a wrong claim that directly contradicts the resolution footer. Provenance is worth keeping; an uncaveated stale conclusion mid-document is a smaller cost to fix than to leave.
- **Suggested resolution:** Append a short inline marker to that bullet (e.g. "— hypothesis; superseded by the Update below: all four were already fixed in 130b720") so the superseded guess cannot be mistaken for the finding.

### [NIT] C3 resolution dresses a paraphrase as a verbatim quote
- **Principle:** 1 (in-line provenance must be accurate)
- **Where:** reviews/2026-05-07-v0.2-setup-and-diagnose.md:95
- **Why it matters:** The C3 bullet ends `...case-insensitive, "do not match a literal string."` — quotation marks imply that phrase is text in the command file. It is not: `git log -S "do not match a literal string" -- commands/` returns only `130b720` as an incidental hit (it removed the word "literal" in the old `chunk X/Y failed` line), and the current `diagnose-loop.md:97` reads "re-checked (case-insensitive) for any line containing `chunk` AND (`fail` OR `error`)". The substance of C3 is correct and correctly attributed; only the quoted artifact is fabricated. Low impact, but C2 and C4 in the same footer use genuinely exact quotes ("never assert positive shape (length, full prefix)" matches line 60 byte-for-byte; ":39" matches), so the inconsistency stands out and erodes the "you can trust these citations" contract.
- **Suggested resolution:** Drop the quote-marks (paraphrase) or replace with the actual file text.

## What was done well
- **The high-stakes claims are real, not theater.** I checked the four concerns against the `130b720` diff directly: C1 (`uv tool show` → `uv tool list --show-with`), C2 (length/prefix asserts dropped → placeholder-only), C3 (literal → `chunk` AND (`fail`/`error`) case-insensitive), and C4's decision logic (`/logout` gated on a 401) all landed in that commit. C2's and C4's footer quotes are exact. The release did **not** manufacture a new false-close on its core findings.
- **C4 closure is genuine.** The Example at :136 previously led with an un-caveated "Resolve auth conflict (`/logout` then re-login)" — a direct contradiction of the :39 rule "Do not recommend `/logout` without an actual 401 report." The diff changes it to "Resolve auth conflict — only if a 401 was reported (`/logout`, then re-login)." The example and the rule now agree; this was the one true residual and it was actually fixed, not papered over.
- **N3's chained attribution is honest.** `fe5ab5c → 97b6c9a` openly records that fe5ab5c's own fix (`upgrade --with`) was itself a bug later corrected to `install --with` in v0.2.2 — it would have been easier to cite a single commit and hide the detour.
- **The v0.2.2 footer's factual assertions check out.** Tag `v0.2.2` exists (`git tag -l`), and the COMMIT_PLAN DoD boxes it claims were reconciled (`:432`, `:434`) are indeed `[x]`. Both nits are legitimately left-as-is with defensible reasoning (provenance preservation, sequencing markers now self-consistent).
- **The drift-detection output stays disciplined.** loop-bootstrap's new drift line (:20) is correctly gated `<only if detected>`, so the "Three lines of context is enough" anti-pattern (:37) is respected on the happy path — the *instructions* got denser but the *user-facing output* did not. The optional drift line only fires when there is real drift to report.
- **Scope is tight and the YAGNI call is right.** No `close_finding.py` was built; the revisit-trigger is documented instead (review-review:41). For four reviews, building atomic-flip tooling would have been the over-engineering the loop is meant to flag. plugin.json is the only non-doc/non-review code touched, and `claude plugin validate .` passes (B0 gate).

## Resolution (2026-05-30)

Status flipped `open` → `closed`, dispositioned in the same session as the review — the layer-1 "same-commit" discipline this very release introduced, applied to itself.

- **CONCERN (N4 misattribution):** ✅ **fixed.** Verified independently — `git show fe5ab5c --stat` = 2 files, no README; `git log -S 'surgical fixes from in-loop review' -- README.md` → `130b720`. The N4 bullet was re-cited `fe5ab5c` → `130b720`, and the footer's opening line now reads "four concerns plus nit N4 in `130b720` / nits N1–N3 in `fe5ab5c`."
- **CONCERN (`/review-review` enumerated from `_index`):** ✅ **fixed.** `review-review.md` line 5 and step 1 now enumerate by globbing `reviews/*.md` and reading each front-matter `status:`, with `_index.md` explicitly demoted to a derived view (regenerated in step 5). The catch-up net no longer has a hole on the drift axis it closes.
- **NIT (`_metrics.md` stale hypothesis):** ✅ **fixed.** The Decision-block parenthetical is marked superseded inline, pointing to the Update that records all four findings were fixed in `130b720`.
- **NIT (C3 quote "not in the file"):** ❌ **rejected — finding is incorrect.** The phrase `do not match a literal string` is verbatim at `commands/setup-graphify.md:82` (`grep -n` confirms), and `git log -S 'do not match a literal string' -- commands/` → `130b720`. The reviewer checked only `diagnose-loop.md:97` (which paraphrases) and missed the `setup-graphify.md` occurrence the footer actually quotes. Quote retained; no change.

**Meta-lesson:** the verification step cut both ways. The independent reviewer caught a real misattribution (N4) that "looked right"; verifying the reviewer's own claims caught one finding (the C3 nit) that was itself unverified. Neither author nor reviewer is exempt from evidence-before-assertion — which is the entire loop.
